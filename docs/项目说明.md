# 汉谜达人项目说明

## 1. 项目简介

汉谜达人是一个面向中文词语识别与连线闯关场景的网页小游戏。玩家需要在一个汉字网格中，通过拖动手势连接横向、纵向或斜向相邻的字，拼出词典中存在的有效词语，并在限定时间内尽可能拿到更高分数。

这个项目保留了经典 Wordament / Boggle 类玩法的核心机制，但把体验重点放在中文语境上：

- 使用中文词库而不是英文单词库。
- 允许玩家通过相邻汉字组合出 2 到 6 字词语。
- 提供更适合移动端滑动操作的界面与反馈。
- 加入每日挑战、练习模式、无尽模式、成就和历史记录。

项目当前是一个前后端同仓的 `Next.js` 应用，既能本地运行，也适合部署为传统 Node 服务或 Docker 容器。

## 2. 核心玩法

### 2.1 基本规则

- 玩家在棋盘中按住并滑动，连接相邻汉字形成路径。
- 仅允许连接相邻格子，包含横向、纵向和斜向。
- 同一条路径中不能重复经过同一个格子。
- 当路径形成的词语存在于词库中时，提交成功并获得分数。
- 找到的词语越长，得分越高；连续命中还会触发连击加成。

### 2.2 游戏模式

- `每日挑战`
  - 固定日期生成固定题目。
  - 默认 4x4 棋盘和普通难度。
  - 同一天所有玩家理论上拿到的是同一题。
- `计时模式`
  - 支持简单、普通、困难三档难度。
  - 支持 3x3、4x4、5x5 三种棋盘规模。
- `练习模式`
  - 不限时，适合熟悉规则或测试词库表现。
- `无尽模式`
  - 固定为 5x5 棋盘。
  - 在 180 秒内连续清盘，系统会自动切到下一盘。

### 2.3 辅助能力

- 历史记录：保存最近 20 局结果。
- 成就系统：根据得分、模式完成情况等解锁成就。
- 本地设置：保存音效开关等用户偏好。
- 每日词语展示：首页展示当天词语、拼音和英文释义。

## 3. 项目特色

- 中文词典驱动：基于 `CC-CEDICT` 词库生成本地可检索词表。
- 前端交互直观：支持拖拽路径、实时高亮、错误抖动、提示起点等反馈。
- 兼容移动端：布局和按钮尺寸适合手机直接游玩。
- 低后端依赖：主要数据都在本地完成，服务端只承担页面渲染和每日词语接口中转。
- 可离线降级：远程释义接口不可用时，自动回退到本地词库释义。

## 4. 技术栈

### 4.1 前端

- `Next.js 16`
  - 使用 App Router。
  - 同时承担页面渲染与 API Route。
- `React 19`
  - 客户端交互、状态管理、拖拽路径展示。
- `TypeScript 5`
  - 为游戏状态、词典结构、配置参数提供类型约束。
- `Tailwind CSS 4`
  - 快速组织页面布局、卡片样式和响应式体验。

### 4.2 后端与运行时

- `Next.js Route Handler`
  - 提供 `/api/daily-word` 接口。
- `Node.js 20+`
  - 推荐的生产运行环境。

### 4.3 数据与构建

- `CC-CEDICT`
  - 作为中文词语基础来源。
- `scripts/build-dictionary.mjs`
  - 负责将压缩词库文件解析为项目使用的 `JSON` 数据。
- `Cookie 持久化`
  - 保存历史记录、成就、每日完成标记、无尽模式统计和用户设置。

## 6. 关键实现说明

### 6.1 词典生成

项目自带 `cedict_ts.u8.gz` 压缩词库文件。构建脚本会：

- 解压原始词库。
- 提取词条中的简体中文字段。
- 过滤出长度在 2 到 6 个汉字之间的词语。
- 输出到 `src/data/dictionary.generated.json`。

对应命令：

```bash
npm run dictionary:build
```

### 6.2 盘面与查词

- 词典在运行时会构建前缀树 `Trie`。
- 棋盘生成模块会根据模式、难度和尺寸创建盘面。
- 每次玩家滑动路径时，前端会根据前缀集合快速判断当前路径是否还有继续尝试的意义。
- 每日挑战通过日期种子保证题目稳定。

### 6.3 计分机制

项目根据词语长度计算基础分，再叠加连击倍率：

- 2 字词：100 分
- 3 字词：250 分
- 4 字词：500 分
- 5 字及以上：1000 分

如果短时间内连续找到词语，会获得额外加成，用来提升节奏感和挑战性。

### 6.4 本地持久化

当前版本将以下数据写入浏览器 Cookie：

- 用户设置
- 历史记录
- 成就状态
- 每日完成标记
- 无尽模式统计

这种方案实现简单，适合单机小游戏，但如果未来需要账号体系、排行榜或跨设备同步，建议迁移到数据库或云端 KV。

### 6.5 每日词语接口

`/api/daily-word` 的处理逻辑是：

1. 先从本地词典中取出当天词语。
2. 尝试请求远程翻译接口，补充拼音和英文释义。
3. 如果远程接口失败，直接回退到本地数据。

这意味着：

- 项目不是强依赖外部接口才能启动。
- 但如果部署环境无法访问外网，首页的每日词语释义可能退回本地默认内容。

## 7. 开发与运行

### 7.1 环境要求

- `Node.js 20` 或更高版本
- `npm 10` 或兼容版本

### 7.2 安装依赖

```bash
npm install
```

可选：复制环境配置示例文件。

```bash
copy .env.example .env.local
```

默认内容如下：

```env
PORT=3000
HOSTNAME=0.0.0.0
```

你可以把 `PORT` 改成任意可用端口，例如 `3010`、`8080`。

### 7.3 本地开发

```bash
npm run dev
```

默认访问：

```text
http://localhost:3000
```

如果你在 `.env.local` 中修改了 `PORT`，这里的端口也会随之变化。

### 7.4 生产构建

```bash
npm run build
npm run start
```

## 8. 适合哪些部署方式

这个项目当前最适合以下几类部署：

- `传统 Node.js 服务器`
  - 最直接，兼容性最好。
- `Docker 容器`
  - 适合云服务器、Kubernetes、宝塔 Docker 管理器等环境。
- `宝塔面板 Node 项目`
  - 适合国内常见运维场景。
- `Cloudflare Workers / Pages`
  - 可做，但更适合在后续补充 `OpenNext` 适配后再正式上线。

其中我最推荐的优先级是：

1. 个人服务器或轻量云主机：`Node.js + Nginx`
2. 希望环境一致：`Docker`
3. 已经在用宝塔：`宝塔 Node 项目部署`
4. 想上边缘平台：`Cloudflare Workers / Pages + OpenNext`
