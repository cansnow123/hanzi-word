# 汉谜达人部署指南

## 1. 部署概览

汉谜达人是一个基于 `Next.js 16` 的全栈项目，包含：

- 页面渲染
- 客户端交互
- 一个服务端接口 `/api/daily-word`

因此它不是纯静态站点，默认更适合部署为：

- `Node.js 服务`
- `Docker 容器`
- `支持 Node 运行时的平台`

如果要部署到 `Cloudflare Workers / Pages` 这类边缘平台，需要引入 `OpenNext` 适配方案。

## 2. 部署前准备

### 2.1 推荐环境

- `Node.js 20+`
- `npm 10+`
- Linux 服务器推荐使用 `Ubuntu 22.04+`

### 2.2 构建前检查

首次部署前建议执行：

```bash
npm install
npm run build
```

如果构建成功，再进入正式部署。

如果需要自定义端口，先复制环境配置示例文件：

```bash
copy .env.example .env.local
```

然后修改：

```env
PORT=3000
HOSTNAME=0.0.0.0
```

### 2.3 端口与外网访问

默认 Node 监听端口建议使用：

```text
3000
```

正式上线通常再通过：

- `Nginx` 反向代理到 80 / 443
- 或平台自带的网关转发

## 3. 方案一：传统 Node.js 部署

这是最稳妥、最直接的方式。

### 3.1 服务器安装环境

安装 Node.js 20 及 npm，上传项目代码后执行：

```bash
npm install
npm run build
```

### 3.2 启动项目

```bash
npm run start
```

默认会启动 Next.js 生产服务。
如果配置了 `.env.local` 或 `.env.production`，启动脚本会自动读取其中的 `PORT` 和 `HOSTNAME`。
另外，`npm run build` 结束后会自动把 `public` 和 `.next/static` 同步到 `.next/standalone`，这是 `standalone` 部署下静态资源正常加载的关键一步。

### 3.3 后台守护

推荐使用 `PM2`：

```bash
npm install -g pm2
pm2 start ecosystem.config.cjs
pm2 save
pm2 startup
```

项目里已经提供：

- [ecosystem.config.cjs](C:/Users/ztab/Documents/New%20project/hanzi-word-connect-open-source/ecosystem.config.cjs)

它会以 `hanzi-word-connect` 作为进程名启动 `npm start`。
当前配置实际执行的是 `node scripts/run-next.mjs start`，并由它读取环境文件后启动 `.next/standalone/server.js`。

### 3.4 Nginx 反向代理示例

```nginx
server {
    listen 80;
    server_name your-domain.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

### 3.5 适用场景

- 云服务器直接部署
- 阿里云 / 腾讯云 / 华为云轻量应用服务器
- 已有 Linux 运维经验

## 4. 方案二：Docker 部署

如果你希望开发、测试、生产环境尽量一致，Docker 是最省心的方案之一。

### 4.1 当前仓库已提供的文件

- [Dockerfile](C:/Users/ztab/Documents/New%20project/hanzi-word-connect-open-source/Dockerfile)
- [.dockerignore](C:/Users/ztab/Documents/New%20project/hanzi-word-connect-open-source/.dockerignore)

同时项目已将 `next.config.ts` 配置为 `output: "standalone"`，便于生成更适合容器运行的精简产物。

### 4.2 构建镜像

```bash
docker build -t hanzi-word-connect:latest .
```

### 4.3 运行容器

```bash
docker run -d --name hanzi-word-connect -p 3000:3000 hanzi-word-connect:latest
```

运行后访问：

```text
http://服务器IP:3000
```

### 4.4 使用环境变量

如需自定义端口，可在运行时传入：

```bash
docker run -d --name hanzi-word-connect -p 3000:3000 -e PORT=3000 hanzi-word-connect:latest
```

也可以通过 `--env-file` 传入：

```bash
docker run -d --name hanzi-word-connect -p 3000:3000 --env-file .env.local hanzi-word-connect:latest
```

### 4.5 配合 Nginx 或 Traefik

生产环境通常建议：

- 容器内部服务监听 `3000`
- 外部由 `Nginx` / `Traefik` 处理 HTTPS、域名和压缩

### 4.6 Docker Compose 示例

```yaml
services:
  hanzi-word-connect:
    build: .
    container_name: hanzi-word-connect
    ports:
      - "3000:3000"
    restart: always
    environment:
      NODE_ENV: production
      PORT: 3000
      HOSTNAME: 0.0.0.0
```

启动命令：

```bash
docker compose up -d
```

### 4.7 适用场景

- 需要标准化部署
- 准备上容器平台
- 后续可能接入 CI/CD

## 5. 方案三：宝塔面板 Node 项目部署

如果你习惯用宝塔，这个项目可以直接按 Node 项目方式部署。

### 5.1 前提

- 宝塔已安装 `Node.js` 管理器
- 服务器已安装 `Node.js 20+`
- 域名已解析到服务器

### 5.2 部署步骤

1. 在宝塔中创建站点，目录指向项目根目录。
2. 上传项目代码到站点目录。
3. 在终端或宝塔文件管理中执行：

```bash
npm install
npm run build
```

如果你需要变更运行端口，也可以先复制并修改环境文件：

```bash
copy .env.example .env.local
```

4. 打开宝塔 `Node 项目管理`。
5. 选择该项目目录，创建 Node 项目。
6. 启动命令填写：

```bash
npm run start
```

7. 运行端口填写：

```text
3000
```

8. 绑定域名，并在网站设置中配置反向代理或直接通过 Node 项目映射。

### 5.3 更稳的做法

如果你更熟悉 PM2，也可以在宝塔服务器里直接执行：

```bash
pm2 start ecosystem.config.cjs
```

再用宝塔 Nginx 站点反向代理到 `127.0.0.1:3000`。

### 5.4 宝塔部署注意事项

- 每次代码更新后，都需要重新执行 `npm run build`。
- 如果你改了依赖，也要重新执行 `npm install`。
- 宝塔的 Node 项目本质上仍然是在跑一个 Node 服务，不是纯静态站点。
- 如果服务器开了安全组或防火墙，要确保反向代理链路可访问。

## 6. 方案四：Cloudflare Workers / Pages

### 6.1 先说结论

这个项目可以部署到 Cloudflare 的边缘平台，但不建议把它当成“零改动直接上传”的项目。

原因是当前仓库是标准的 `Next.js` 全栈结构，包含：

- App Router
- Route Handler
- 服务端 `fetch`

因此更适合通过 `OpenNext` 适配到 Cloudflare Workers。

### 6.2 当前可行路线

根据 Cloudflare 官方文档，Next.js 项目可以通过 `OpenNext` 适配器部署到 Workers，且大多数特性都可用，包括：

- App Router
- Route Handlers
- SSR
- ISR
- Server Actions
- Middleware

但文档也明确提到，`Node.js in Middleware` 这类能力仍有不支持项。

### 6.3 适配时通常需要做的事情

1. 安装 `@opennextjs/cloudflare`
2. 安装 `wrangler`
3. 增加 `wrangler.jsonc`
4. 增加 `open-next.config.ts`
5. 使用 Cloudflare 提供的脚本进行 `build / preview / deploy`

### 6.4 为什么这里不建议直接给仓库硬塞 Cloudflare 配置

因为当前项目并没有安装这些依赖，也还没有做过实际联调验证。直接加配置文件但不装依赖，容易让仓库看起来“支持 Cloudflare”，实际却无法一键部署。

更稳妥的做法是：

- 先按 `Node` 或 `Docker` 上线首版
- 如果后续确实要上 Cloudflare，再单独补一轮适配和联调

### 6.5 什么时候适合上 Cloudflare

- 希望使用边缘节点降低全球访问延迟
- 后续打算接入 Cloudflare KV / R2 / D1
- 团队已经熟悉 Wrangler 和 Workers 生态

## 7. 关于 Worker 方案的补充建议

如果你提到的 `worker` 不是特指 Cloudflare，而是泛指“后台常驻服务”或“守护进程”，那么当前项目通常不需要额外拆出独立 Worker 进程，因为：

- 游戏逻辑主要在前端完成
- 服务端没有复杂队列任务
- 每日词语接口只是轻量级读取和远程转发

也就是说，当前阶段：

- `PM2` 就足够承担守护任务
- 不需要额外设计消息队列 Worker

## 8. 环境变量与外部依赖

当前仓库没有强制要求额外环境变量才能运行。

但有一个外部依赖需要注意：

- `/api/daily-word` 会访问远程翻译接口获取拼音和英文释义

如果部署环境无法访问这个接口：

- 项目仍然可以正常运行
- 只是首页展示的每日词语信息会回退到本地默认内容

## 9. 推荐上线顺序

如果你是第一次部署这个项目，我建议按下面顺序选择：

1. `Node.js + PM2 + Nginx`
   - 最直观，排错最方便。
2. `Docker + Nginx`
   - 最适合标准化交付。
3. `宝塔 Node 项目`
   - 适合国内常见运维习惯。
4. `Cloudflare Workers / Pages`
   - 适合作为第二阶段优化。

## 10. 官方资料参考

以下是我核对过的官方资料，部署时可以一并参考：

- Next.js 官方部署文档：[Getting Started: Deploying](https://nextjs.org/docs/app/getting-started/deploying)
- Next.js 官方自托管文档：[Guides: Self-Hosting](https://nextjs.org/docs/app/guides/self-hosting)
- Cloudflare 官方 Next.js 文档：[Next.js · Cloudflare Workers docs](https://developers.cloudflare.com/workers/framework-guides/web-apps/nextjs/)

基于这些官方资料，可以确认几件事：

- Next.js 当前官方支持部署为 `Node.js server`、`Docker container`、`static export` 或适配到其他平台。
- Docker 仍然是官方明确支持的一等部署方式。
- Cloudflare 对 Next.js 的推荐路径是通过 `OpenNext` 适配到 Workers。

## 11. 我对本项目的最终建议

对汉谜达人这个项目来说，当前最务实的上线方式是：

- 开发环境：`npm run dev`
- 测试 / 生产：`Docker` 或 `Node.js + PM2`
- 面板运维：`宝塔 Node 项目`
- 边缘化部署：后续再做 `Cloudflare OpenNext` 适配

这样能在不增加过多复杂度的前提下，把项目快速、稳定地上线。
